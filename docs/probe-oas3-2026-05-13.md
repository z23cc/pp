## API surface / ergonomics

- Spec deserialization exists: `oas3::from_yaml(...)` with `yaml-spec`; JSON via serde (`serde_json::from_str::<oas3::Spec>` worked in the probe).
- Walking paths × methods is straightforward but shape differs: `Spec.paths` is `Option<Map<String, PathItem>>`, not a required paths object. This already required unwrapping in the probe.
- `Operation` has the expected fields: `operation_id`, `parameters`, `request_body`, `responses`, `security`.
- Security schemes live under `components.security_schemes: Map<String, ObjectOrReference<SecurityScheme>>`.
- Reference analogue exists: `ObjectOrReference<T>` at `src/spec/ref.rs:18`, pattern-matchable like openapiv3's `ReferenceOr<T>`, but all call sites need renaming and shape checks.
- Types are `Clone` and `Serialize`-able; source derives show `#[derive(Debug, Clone, ..., Deserialize, Serialize)]` on `Operation`, `Response`, `RequestBody`, `SecurityScheme`, etc.
- Vendor extensions are first-class flattened fields: e.g. `Operation.extensions: Map<String, serde_json::Value>` and README says typed OpenAPI maps and specification extensions preserve order.

## Migration cost estimate

High cost for no parser win.

- `pp/src/spec/mod.rs`: direct parser/type impact around imports, `SpecWithFacts.api`, `parse`, `count_operations`, `derive_auth_kind`, `auth_kind_for_scheme`, and tests. Roughly 120–180 LOC touched.
- `pp/src/spec/normalize.rs`: much larger rewrite. It mutates openapiv3-specific `SchemaKind`, `Type`, `ArrayType`, `ObjectType`, `StatusCode`, `ReferenceOr`, `MediaType`, `RequestBody`, `Response`, and path item shapes. Roughly 300–500 LOC touched.
- Total rough port: **450–700 LOC** plus test fixture churn. Biggest breaking change is schema representation, not path walking.
- Additional hidden cost: `pp` feeds `openapiv3::OpenAPI` to progenitor-adjacent code today; swapping parser types may require converting back to JSON/openapiv3 or changing downstream assumptions.

## Recommendation

**NO.** Do not replace `openapiv3` with `oas3` for this bug class.

`oas3` is maintained and ergonomic enough for basic walking, but empirically it rejects both failure specs before normalization, exactly where `pp` needs tolerance. The cheaper and more targeted path is a YAML/JSON pre-normalization pass before typed deserialization: coerce/drop non-scalar tag descriptions and clamp/remove out-of-range numeric bounds.
