## Findings (prioritized)

### P0 — blockers
- OpenAPI 3.1 downgrade does not make the current OpenAI spec parseable. `pp inspect /tmp/pp-dogfood/openai.yaml` applied 2 transforms, then failed on `components.schemas.CreateChatCompletionRequest.allOf[1].properties.seed.minimum` because `-9223372036854776000` could not be represented as the expected value at line 42624.
- DigitalOcean's vendor spec fails before inspection/generation because `tags[0].description` is a YAML map, but the parser expects a string at line 25. This blocks a well-known bearer-auth infra API entirely.

### P1 — UX warts
- API-key auth detection missed Interzoid's required `license` query parameter. `pp inspect` reported `auth_kind: { kind: "none" }`, while generated help requires `--license <license>` and describes it as the API key.
- The generated binary name `interzoid-get-weather-city-api` is long and product-title-derived. It is technically valid but awkward as a daily CLI name.
- Runtime HTTP errors omit the response body. Fake invocation returned status 403 and headers only, losing the plain-text body that likely explains the invalid license.
- Generated release build succeeds but emits unused-code/import warnings in generated code: `anyhow::Context as _`, `base64::Engine`, unused `request`, and unread `auth`.

### P2 — nice-to-haves
- Root help is generic: `Generated API CLI`; it does not include API name, auth hint, base URL, or a next-command example.
- Normalization messages are useful but not surfaced in `inspect` JSON for the Interzoid case. `generate` reported: kept 200, dropped 400, 402, 403, 404, 405, 500.
- Required query parameters are represented faithfully, but the UX makes API keys feel like ordinary endpoint arguments rather than credentials that could come from env/config.

## Comparison: what's different from petstore/Plausible/PokeAPI

- Current SaaS specs include JSON Schema numeric edge cases that exceed parser expectations after OpenAPI 3.1 downgrade.
- Vendor metadata can contain rich Markdown/object descriptions where `pp` currently expects scalar strings.
- API-key auth may be represented as a required operation parameter rather than a formal `securitySchemes` entry.
- Real error responses may be plain text and valuable, but generated clients can discard the body.
- Product-title-derived names become much longer on real catalog-style APIs.

## Recommended next plan

- Make spec loading more tolerant before generation: normalize rich tag descriptions and large numeric bounds, especially after 3.1 downgrade.
- Improve auth inference for API-key-like required parameters such as `license`, `api_key`, and `appid`, then offer env/config support.
- Improve generated CLI diagnostics: include API name/auth hints in help and preserve response bodies for non-2xx errors.
